# MiroThinker 技术文档版本对比分析

> 对比对象：GitHub 原始 README.md vs 当前 PROJECT_TECH_DOC.md

---

## 一、总体定位差异

| 维度 | 原始 README.md (GitHub 版) | 当前 PROJECT_TECH_DOC.md |
|------|---------------------------|-------------------------|
| **目标受众** | 终端用户、初次接触的开发者 | 技术负责人、核心开发者、架构评审 |
| **文档定位** | 快速入门指南 (Quick Start Guide) | 完整技术白皮书 (Technical Specification) |
| **阅读目的** | "怎么安装、怎么调用" | "怎么实现的、为什么这样设计、成本多少" |
| **篇幅** | ~230 行，约 5.8 KB | ~550+ 行，约 20+ KB |

**核心差异**：原始版本是"操作手册"，当前版本是"设计蓝图"。

## 三、逐项深度对比

### 3.1 架构描述

| 方面 | 原始 README | 当前文档 |
|------|------------|---------|
| **架构图** | 无 | 有 ASCII 三层架构图 |
| **请求生命周期** | 无 | 12 步完整流程，含数据流方向 |
| **分层说明** | 无 | 接入层 / 编排层 / 能力层，每层职责清晰 |
| **模块交互** | 无 | 主循环 → Pre-search → Adaptive Chains → Quality → Critic 的完整链路 |

**评价**：原始版本完全没有架构层面的描述，读者无法从文档中理解系统是如何协作的。当前文档补足了这一关键缺失。

---

### 3.2 模块功能说明

| 方面 | 原始 README | 当前文档 |
|------|------------|---------|
| **项目结构** | 只有目录树，无文件职责 | 每个核心文件都有"职责 + 关键函数 + 实现逻辑"说明 |
| **agent.py** | 仅标注 "Research agent core" | 详细说明 run() / call_llm() / _critic_review() / _pre_search() / AgentState / classify_and_route() |
| **quality.py** | 仅标注 "Quality enhancement modules" | 拆解为 8 项通用检查 + 4 项金融检查，每项说明评分逻辑 |
| **domain 系统** | 完全未提及 | 专设 domains/ 章节，说明注册表、自适应链组合、金融域 6 大思维链 |

**评价**：原始版本的项目结构只回答了"有哪些文件"，当前文档回答了"每个文件做什么、怎么做、为什么这么做"。

### 3.3 核心机制透明度

| 机制 | 原始 README | 当前文档 |
|------|------------|---------|
| **Agent 循环** | 仅提到 "Multi-turn autonomous research" | 提供完整 while 循环伪代码，含动态上下文窗口、分层模型选择、工具解析、早停逻辑 |
| **反循环** | 未提及 | 三层防护 + Jaccard 相似度公式 + 警告注入 |

### 3.4 成本分析

| 方面 | 原始 README | 当前文档 |
|------|------------|---------|
| **计量单位** | Credits（百炼平台积分） | 人民币（¥） |
| **精度** | Tier 级别粗估（50/500/600 Credits） | 阶段级别精算（分类 ¥0.005 + 搜索 29×¥0.003 + 合成 ¥0.15） |
| **优化对比** | 无 | 优化前后 5 项指标对比表，总成本从 ¥1.7 → ¥0.25 |
| **月度估算** | "166-2000 tasks/month with ¥198 package" | 日均 50 次 × 30 天 = ¥375/月，明确节省 ¥2,175/月 |
| **模型差异** | 未区分模型成本差异 | 明确区分 qwen-turbo / qwen-flash / qwen3.6-plus 的单价差异 |

**评价**：原始版本的 Credits 估算是面向平台用户的，当前版本的人民币精算是面向商业决策者的，可直接用于预算审批。

| 方面 | 原始 README | 当前文档 |

## 四、优劣势总结

### 4.1 原始 README 的优势

1. **上手门槛低**：新用户可以在 5 分钟内完成安装和首次调用
2. **信息密度高**：快速了解项目能做什么，适合技术选型阶段
3. **标准化程度高**：包含 Docker、CI/CD、Makefile 等工程化内容
4. **多语言友好**：英文为主，适合国际开发者

### 4.2 原始 README 的劣势

1. **架构黑盒**：无法从中理解系统内部如何协作
2. **机制缺失**：没有解释任何核心算法或设计决策
3. **成本模糊**：Credits 估算无法直接换算为运营成本
4. **维护困难**：新开发者接手后无法通过 README 理解代码逻辑
5. **领域系统隐身**：domains/ 目录下的复杂领域适配逻辑完全没有提及
6. **质量系统空洞**："quality enhancement" 是功能名，不是实现说明

### 4.3 当前技术文档的优势

1. **架构可视化**：三层架构图 + 12 步生命周期，一目了然
2. **机制白盒化**：核心循环、反循环、分层模型都有伪代码或流程图
3. **成本可量化**：精确到每次 API 调用的人民币成本，支持商业决策
4. **模块职责清晰**：每个核心文件都有"做什么 + 怎么做"的说明
5. **优化有记录**：已完成和待完成的优化都明确列出，便于版本管理
6. **问题可定位**：明确列出当前最大技术债务（工具调用格式不稳定）
7. **领域系统显性化**：金融域的 6 大思维链、子类型补充链都有详细说明

### 4.4 当前技术文档的劣势

1. **上手门槛高**：不适合只想快速试用的用户，需要有一定技术背景
2. **篇幅较长**：20+ KB 的文档需要 30~60 分钟完整阅读
3. **缺少部署细节**：相比原始 README，缺少 Docker Compose、Makefile 等工程化内容（这些内容更适合保留在 README 中）
4. **无交互示例**：缺少 API 调用的 curl 示例和响应样例

## 五、建议的文档分工

两者并非替代关系，而是互补关系：

| 文档 | 职责 | 受众 | 存放位置 |
|------|------|------|---------|
| **README.md** | 快速入门、安装部署、API 调用示例、工程规范 | 新用户、DevOps、测试人员 | 项目根目录 |
| **PROJECT_TECH_DOC.md** | 架构设计、模块详解、核心机制、成本分析、优化路线 | 技术负责人、核心开发者、架构评审 | 项目根目录 / docs/ |

**建议**：保留 README.md 的 Quick Start 和 Deployment 章节，将 Architecture 和 Implementation 章节替换为指向 PROJECT_TECH_DOC.md 的链接，避免维护两份重复内容。

---

## 六、关键结论

1. **原始 README 是产品手册**：回答了 "What" 和 "How to use"
2. **当前文档是工程蓝图**：回答了 "How it works"、"Why this design"、"How much it costs"、"What's next"
3. **两者缺一不可**：没有 README，新用户无法上手；没有技术文档，团队无法维护和演进
4. **当前最大改进**：将质量系统、领域系统、成本模型从黑盒功能还原为白盒机制，这是从 Demo 级项目向生产级系统转型的关键标志
