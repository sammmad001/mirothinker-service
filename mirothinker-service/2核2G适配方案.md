# 2核2G 云服务器适配方案

> 评估日期: 2026-05-11
> 服务器配置: 2核 CPU, 2GB 内存
> 状态: ✅ 可以部署，但需优化配置

---

## 一、可行性分析

### 2核2G 能否运行？

**结论: ✅ 可以，但有条件**

| 组件 | 最小内存需求 | 说明 |
|------|-------------|------|
| Python 进程 | ~200-300MB | FastAPI + 依赖 |
| LLM 调用 | 0MB (云端) | 调用百炼 API，不占用本地资源 |
| 搜索/抓取 | ~50-100MB | httpx 异步客户端 |
| 操作系统 | ~300-500MB | Linux 系统基础 |
| 其他应用 | 视情况而定 | 需预留 |
| **总计** | **~600-900MB** | 不含其他应用 |

### 关键约束

- ✅ **可行**: MiroThinker 本身是 **I/O 密集型** 应用，主要消耗在等待 API 响应
- ⚠️ **限制**: 并发任务数需限制 (建议 1-2 个同时任务)
- ⚠️ **注意**: 需合理分配内存给其他应用

## 二、资源配置优化

### 1. 调整 Worker 数量

**原配置**: `--workers 2`  
**优化后**: `--workers 1`

原因:
- 每个 worker 约 200-300MB 内存
- 2 个 worker = 400-600MB，对 2G 服务器偏多
- 1 个 worker 足够处理请求 (I/O 密集型)

### 2. 限制并发任务数

**原配置**: 无限制  
**优化后**: 最多 2 个同时进行的研究任务

原因:
- 每个研究任务在内存中维护对话状态
- 过多并发任务会导致内存压力

### 3. 调整上下文保持

**原配置**: `context_keep: int = 5` (默认)  
**建议**: 保持默认或降低到 3

原因:
- 减少内存中的对话历史
- 对研究质量影响不大

## 四、优化后的部署配置

### Docker Compose 优化版

```yaml
version: '3.8'

services:
  mirothinker:
    build: .
    container_name: mirothinker-service
    ports:
      - "8000:8000"
    env_file:
      - .env
    restart: unless-stopped
    # 资源限制 (适配 2核2G)
    deploy:
      resources:
        limits:
          cpus: '1.0'        # 最多使用 1 个 CPU 核心
          memory: 512M       # 限制最大内存 512MB
        reservations:
          cpus: '0.5'        # 预留 0.5 个 CPU 核心
          memory: 256M       # 预留 256MB 内存
    # 单 worker 模式
    command: >
      uvicorn main:app
      --host 0.0.0.0
      --port 8000
      --workers 1
      --limit-concurrency 3    # 限制并发连接
      --backlog 20             # 限制请求队列
    # 日志限制 (节省磁盘)
    logging:
      driver: "json-file"
      options:
        max-size: "20m"
        max-file: "2"
    # 健康检查
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]
      interval: 60s           # 降低检查频率
      timeout: 10s
      retries: 3
      start_period: 15s
```

### 直接部署优化命令

```bash
# 启动单 worker，限制并发
nohup python3 -m uvicorn main:app \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 1 \
    --limit-concurrency 3 \
    --backlog 20 \
    > /var/log/mirothinker.log 2>&1 &
```

```bash

docker stats

docker-compose restart

# 3. 如持续内存泄漏，检查日志
docker-compose logs --tail=100
```

### 服务响应慢时

```bash
# 1. 检查 CPU

# 2. 检查是否有其他应用占用资源
ps aux --sort=-%mem | head -10

# 3. 考虑错峰使用 (如其他应用只在白天使用)
```

## 十、结论

### ✅ 2核2G 可以部署 MiroThinker

**适用场景**:
- ✅ 个人使用或小团队 (日查询 < 20 次)
- ✅ 与其他轻量应用共存
- ✅ 研究型应用 (非实时高并发)

**不适用场景**:
- ❌ 高并发商用 (日查询 > 100 次)
- ❌ 与其他重型应用同时运行
- ❌ 需要亚秒级响应的场景

**建议**:
1. 使用 Docker Compose 部署 (便于资源控制)
2. 限制并发任务数为 2
3. 监控内存使用，设置告警
4. 如预算允许，建议升级到 4G 内存
