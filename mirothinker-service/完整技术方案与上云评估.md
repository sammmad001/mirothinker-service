## 3. 核心模块设计

### 3.1 三阶段研究流水线

```
Phase 1: 浅层研究 (目标: 20-25 信息源)
├── 生成 5 个搜索查询 (并行)
├── 执行 5 次 Brave Search → 25-30 个搜索结果
├── 筛选 8 个关键 URL
├── 执行 8 次 Jina 抓取 (并行)
├── LLM 初步综合 (qwen-plus, 2 次调用)
├── 信息充足度评估 (qwen-flash)
│
├── 充足 (25%) → 输出
│
└── 不足 → Phase 2

Phase 2: 聚焦深入 (目标: 10-15 信息源)
├── 识别 Phase 1 的知识 gap (qwen-plus)
├── 生成 3 个针对性搜索查询
├── 执行 3 次 Brave Search
├── 筛选 6 个深度 URL
├── 执行 6 次 Jina 抓取
├── LLM 深度分析 (qwen-plus, 2 次调用)
├── 再评估
│
├── 充足 → 输出
│
└── 不足 → Phase 3

Phase 3: 交叉验证 (目标: 2-3 信息源)
├── 提取 3 个关键主张 (qwen-plus)
├── 每个主张找 1 个独立来源验证
├── LLM 综合判断 (qwen-plus, 2 次调用)
└── 最终输出 (含引用来源)
```

### 3.2 Tier 路由分类器

```python
ROUTING_PROMPT = """
Classify this query into exactly one tier:

TIER_1: Factual question, answerable in 1-2 sentences
Examples: "What is the capital of France?", "Who created Python?"

TIER_2: Comparison, analysis, or explanation needed
Examples: "Compare Python vs Rust for web development"

TIER_3: Deep research requiring multiple sources and synthesis
Examples: "Analyze the economic impact of AI on healthcare by 2030"

Query: {query}
Return ONLY: TIER_1, TIER_2, or TIER_3
"""
```

### 3.3 缓存体系

| 缓存层 | 键生成方式 | TTL | 命中率预估 |
|--------|-----------|-----|-----------|
| 查询缓存 (精确) | MD5(query) | 24h | 5-10% |
| 查询缓存 (相似) | 编辑距离 > 0.85 | 24h | 5-10% |
| LLM 响应缓存 | MD5(messages) | 1h | 10-15% |

## 4. 上云方案评估

### 4.1 阿里云资源配置

| 资源 | 推荐规格 | 月成本 | 说明 |
|------|----------|--------|------|
| **ECS 实例** | ecs.t6-c1m2.large (2vCPU/4GB) | ¥150-200 | 轻量应用，够用 |
| **ECS 实例** (推荐) | ecs.c7.large (2vCPU/4GB) | ¥250-300 | 计算优化型，更稳定 |
| **系统盘** | 40GB SSD | ¥20 | 基础存储 |
| **带宽** | 5Mbps 按量 | ¥50-80 | 按流量计费更省 |
| **SSL 证书** | Let's Encrypt (免费) | ¥0 | 自动续期 |

### 4.2 服务器资源需求评估

| 指标 | 需求 | 2vCPU/4GB 是否满足 |
|------|------|-------------------|
| CPU 占用 | FastAPI 2 workers + Docker 沙箱 | ✓ 峰值 60-70% |
| 内存占用 | Python ~500MB + Docker ~1GB | ✓ 剩余 ~2GB |
| 磁盘 | 代码 + 日志 + Docker 镜像 | ✓ 40GB 足够 |
| 带宽 | 静态资源 + API 响应 | ✓ 5Mbps 足够 |
| 并发 | 同时 2-3 个研究任务 | ✓ 可接受 |

**结论**: 2vCPU/4GB 实例可支持日均 5-10 次研究任务，满足当前需求。

### 4.3 部署架构选择

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **单 ECS 直部署** (推荐初期) | 简单、成本低、易维护 | 无高可用，单点故障 | ⭐⭐⭐⭐ (初期) |
| ECS + SLB 双机 | 高可用，自动故障切换 | 成本翻倍 | ⭐⭐⭐ (日活 > 20) |
| 容器服务 ACK | 弹性伸缩，易扩展 | 运维复杂，成本高 | ⭐⭐⭐ (规模化后) |
| 函数计算 + API 网关 | 按量付费，零运维 | 冷启动慢，不适合长任务 | ⭐ (不推荐) |

### 6.2 稳定性评估 (Stability)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 服务可用性 | ⭐⭐⭐⭐ | 单 ECS 无 SLA 保障，预计 99%+ |

**提升稳定性的建议**:

2. **Nginx 限流**: 防止突发流量打满服务
3. **超时设置**: API 请求 10 分钟超时，防止阻塞
4. **日志轮转**: 防止日志占满磁盘
5. **健康检查**: 定时检测服务状态
