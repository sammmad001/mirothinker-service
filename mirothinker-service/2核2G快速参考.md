# 2核2G 服务器适配总结

> ✅ **结论: 可以部署，已与现有应用共存方案**

---

## 一句话回答

**可以！** MiroThinker 已针对 2核2G 优化，限制为单 Worker + 最多 2 个并发任务，内存占用控制在 **300-400MB**，可以与其他应用安全共存。

---

## 关键优化

| 项目 | 原配置 | 2核2G 优化后 | 说明 |
|------|--------|-------------|------|
| Worker 数量 | 2 | **1** | 减少内存占用 |
| 并发任务 | 无限制 | **最多 2 个** | 防止内存溢出 |
| Docker 内存限制 | 4GB | **512MB** | 硬性限制 |
| Docker CPU 限制 | 2 核 | **1 核** | 避免抢资源 |
| 健康检查间隔 | 30s | **60s** | 降低开销 |
| 日志大小 | 50MB × 3 | **20MB × 2** | 节省磁盘 |

---

## 资源占用预期

```
MiroThinker 运行状态 (2核2G):

├── 内存: 300-400MB (正常运行)
├── CPU: 10-30% (空闲时 <10%)
└── 磁盘: ~100MB (代码 + 日志)

剩余可用资源:
├── 内存: ~1.2-1.4GB (供其他应用)
├── CPU: ~1 核 (供其他应用)
└── 完全不影响其他服务运行
```

---

## 与其他应用共存

### ✅ 安全共存的应用

| 应用类型 | 内存占用 | 是否冲突 |
|----------|---------|----------|
| Nginx (静态网站) | ~50MB | ✅ 不冲突 |
| MySQL (小型) | ~200MB | ✅ 可以 |
| Redis (缓存) | ~50MB | ✅ 可以 |
| Node.js 应用 | ~100-200MB | ✅ 可以 |
| 其他 Python 应用 | ~200MB | ✅ 可以 (不同时高峰) |

### ⚠️ 需要注意的情况

| 情况 | 建议 |
|------|------|
| 服务器已有应用占用 >1GB 内存 | 建议错峰使用或升级配置 |
| 多个应用同时高负载 | 设置 CPU 亲和性或 cgroups |
| 数据库占用大量内存 | 限制 MySQL `innodb_buffer_pool_size` |

## 一键部署

```bash
# 上传到服务器
scp -r mirothinker-service/ root@YOUR_SERVER:/opt/

# SSH 登录
ssh root@YOUR_SERVER
cd /opt/mirothinker-service

# 运行优化部署脚本
chmod +x deploy-2core-2g.sh
./deploy-2core-2g.sh
```

脚本会自动:
1. 检测系统资源
2. 使用优化配置 (单 Worker, 512MB 内存限制)
3. 启动服务并验证
4. 显示管理命令

---

## 验证命令

```bash
# 1. 健康检查
curl http://localhost:8000/api/health

# 2. 查看系统状态 (含内存、并发信息)
curl http://localhost:8000/api/status

# 3. Docker 模式下查看资源使用
docker stats mirothinker-service

# 4. 直接模式下查看进程内存
ps aux | grep uvicorn
```

## 故障处理

### 内存不足

```bash
# 重启 MiroThinker 释放内存
docker-compose restart  # Docker 模式
pkill -f uvicorn && nohup python3 -m uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1 > /var/log/mirothinker.log 2>&1 &  # 直接模式
```

### 服务无响应

```bash
# 检查日志
docker-compose logs --tail=50  # Docker
tail -50 /var/log/mirothinker.log  # 直接

# 重启服务
docker-compose restart
```
