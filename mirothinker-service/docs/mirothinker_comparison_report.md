## 一、当前技术方案概览

### 1.1 技术架构

```
┌──────────────────────────────────────────────────┐
│              MiroThinker 当前架构                │
├──────────────────────────────────────────────────┤
│                                                  │
│  搜索层: DuckDuckGo (ddgs) + Trafilatura          │
│  ├── 优势: 零成本、无需 API Key                   │
│  └── 劣势: 搜索质量不稳定、无结构化验证            │
│                                                  │
│  LLM层: 阿里云百炼 (qwen-flash/plus)              │
│  ├── 优势: 国内合规、成本低                       │
│  └── 劣势: 相比国际顶级模型推理能力有限           │
│                                                  │
│  质量层: FixedChannelSearch + SourceCredibility   │
│  ├── 信源可信度评分: 基础权重法 (40% 基础 + ...)  │
│  └── 矛盾检测: 数值差异 > 20% 触发                │
│                                                  │
│  持久化: SQLite (SessionStore)                    │
│  └── 无知识图谱，缺少实体关联能力                 │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 1.2 核心功能模块

| 功能模块 | 当前实现 | 说明 |
|---------|---------|------|
| 多阶段研究 | Three-Phase Loop | P1浅层 → P2聚焦 → P3交叉验证 |
| Tier路由 | 基础分类 | qwen-flash 分类为 TIER_1/2/3 |
| 质量保障 | 7维检查 | 来源数、引用完整性、数据支撑等 |
| 信源分级 | 预定义权重 | A/B/C/D 四级，依赖人工配置 |
| 矛盾检测 | 阈值触发 | 仅数值矛盾 >20%，定性检测弱 |

### 1.3 当前方案的主要问题

| 问题类别 | 具体表现 | 影响程度 |
|---------|---------|---------|
| **准确性** | 缺少自动化来源质量评估模型 | 高 |
| **准确性** | 矛盾检测仅限数值，定性冲突未覆盖 | 中 |
| **可追溯性** | 无引用归因的内联验证机制 | 高 |
| **可追溯性** | 无 URL 可用性自动检查 | 中 |
| **时效性** | 无增量更新机制，重复研究浪费资源 | 高 |
| **时效性** | 无网页更新监控，无法感知内容变化 | 中 |
| **架构** | 无知识图谱，缺少实体关联能力 | 中 |

### 2.1 验证代理架构

#### 2.1.1 Step-DeepResearch 验证代理架构

**来源**: arXiv:2512.20491

**核心架构**:

```
┌─────────────────────────────────────────────────────────────┐
│  Step-DeepResearch 技术架构                                  │
├─────────────────────────────────────────────────────────────┤

│  │  ├── 执行搜索 (DuckDuckGo/Brave)                      │   │
│  │  ├── 抓取页面内容 (Trafilatura/Playwright)            │   │
│  │  └── 综合推理                                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                         │                                   │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          Verify Agent (验证代理)                     │   │
│  │  ├── 事实核查: 逐句验证声明                          │   │
│  │  ├── 引用归因: URL 可访问性检查                       │   │

│  │  └── 返回验证报告                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                         │                                   │
│                         ▼                                   │
│           验证失败? ──是──> 返回 Research Agent 重做        │
│              │                                           │
│              否                                          │
│              ▼                                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          Final Report (最终报告)                     │   │
│  │  ├── 已验证的事实                                    │   │

│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键技术创新**:

| 技术 | 描述 | 效果 |
|------|------|------|
| **逐句验证** | 不是整体评分，而是逐句定位 substantive inconsistencies | 精细化质量控制 |
| **Pointwise 评分** | 每个不一致导致扣分，而非整体评级 | 更精确的评估 |
| **Citation Traceability** | 确保每个声明都链接到对应来源 | 引用完整性 +30% |
| **Inline Citation** | 内联标注 [1][2] 格式 | 可追溯性增强 |
| **One-to-One Mapping** | 引用与参考条目一一映射，防止孤立引用 | 格式规范化 |

**Agent-Ensemble-Judge**:

```
┌─────────────────────────────────────────────────────────────┐
│  多模型联合评判 (Agent-Ensemble-Judge)                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Gemini 2.5 Pro ─────┐                                      │
│                     ├──> Average ──> 评判结果               │
│  GPT-5 ─────────────┘                                      │
│                                                             │
│  优势:                                                      │
│  ├── 减少单模型偏见                                        │
│  ├── 与人类判断高度一致                                    │
│  └── 避免大模型变异性 (large variability)                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**与 MiroThinker 对比**:

| 维度 | MiroThinker | Step-DeepResearch | 差距 |
|------|-------------|------------------|------|
| 验证机制 | 无独立验证代理 | Verify Agent 独立验证 | 缺少核心组件 |
| 引用归因 | 基本引用列表 | Citation Traceability | 2代差距 |
| 事实核查 | 简单阈值检查 | 逐句验证 + 多模型评判 | 3代差距 |
| 冲突处理 | 数值 >20% | Claim-URL 配对验证 | 2代差距 |

### 2.2 当前 RAG 应用分析

#### 2.2.1 当前 RAG 应用环节

| 模块 | 文件位置 | RAG 类型 | 实现方式 | 状态 |
|------|---------|---------|---------|------|
| **搜索缓存** | `search.py` (SearchCache) | 基础检索缓存 | 内存 dict，基于 query hash，TTL 1小时 | ✅ 已有 |
| **预搜索** | `agent.py` (_pre_search) | 时序感知检索 | DuckDuckGo + 2025/2026 显式年份搜索 | ✅ 已有 |
| **ResearchMemory** | `research_memory.py` | 结构化知识存储 | 跨轮次知识积累，非传统 RAG | ✅ 已有 |
| **MemoryExtractor** | `research_memory.py` | 知识提取 | qwen-flash 提取结构化知识 | ✅ 已有 |
| **固定渠道搜索** | `quality.py` (FixedChannelSearch) | 分层检索 | L1核心→L2扩展→L3通用 | ✅ 已有 |
| **信源评分** | `quality.py` (SourceCredibilityScorer) | 评分型 RAG | 预定义权重 + 内容质量 + 时效性 | ✅ 已有 |
| **矛盾检测** | `quality.py` (ContradictionDetector) | 冲突检测 | 数值/定性冲突阈值检测 | ✅ 已有 |

**当前 RAG 特点**:
- ✅ 搜索层有缓存机制
- ✅ 有预搜索 + 时序感知
- ✅ 有结构化知识存储 (ResearchMemory)
- ❌ **无向量检索** (无 embedding 模型)
- ❌ **无混合检索** (无 BM25 + vector 组合)
- ❌ **无重排序** (无 cross-encoder)
- ❌ **无知识图谱** (无 GraphRAG)

#### 2.2.2 所需 RAG 能力与现有差距

| RAG 能力 | 当前状态 | 差距 | 优先级 |

# backend/src/services/embedding.py
from openai import OpenAI
from src.core.config import settings

class BailianEmbedder:
    """百炼 Embedding 服务 (OpenAI 兼容接口)"""
    
    def __init__(self):
        self.client = OpenAI(
            api_key=settings.DASHSCOPE_API_KEY,
            base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"
        )
        self.model = "text-embedding-v4"
    
    def embed(self, text: str) -> list[float]:
        response = self.client.embeddings.create(
            model=self.model,
            input=text
        )
        return response.data[0].embedding
    
    def embed_batch(self, texts: list[str]) -> list[list[float]]:
        response = self.client.embeddings.create(
            model=self.model,
            input=texts
        )
        return [item.embedding for item in response.data]

**ChromaDB + BM25 混合检索**:

```python
# backend/src/services/hybrid_retrieval.py
import chromadb
from rank_bm25 import BM25Okapi
from .embedding import BailianEmbedder

class HybridRetriever:
    """BM25 + ChromaDB 混合检索"""
    
    def __init__(self):
        self.chroma = chromadb.Client()
        self.collection = self.chroma.create_collection("research")
        self.embedder = BailianEmbedder()
        self.bm25 = None
        self.corpus = []
    
    def index(self, texts: list[str]):
        # ChromaDB 向量索引
        embeddings = self.embedder.embed_batch(texts)
        self.collection.add(embeddings=embeddings, documents=texts)
        
        # BM25 索引
        tokenized = [t.split() for t in texts]
        self.bm25 = BM25Okapi(tokenized)
        self.corpus = texts
    
    def search(self, query: str, top_k: int = 10) -> list[dict]:
        # BM25 得分
        bm25_scores = self.bm25.get_scores(query.split())
        bm25_results = sorted(enumerate(bm25_scores), key=lambda x: x[1], reverse=True)[:top_k]
        
        # 向量检索
        query_embedding = self.embedder.embed(query)
        vector_results = self.collection.query(query_embeddings=[query_embedding], n_results=top_k)
        
        # 混合融合 (BM25 0.3 + Vector 0.7)
        fusion_scores = {}
        for i, (idx, score) in enumerate(bm25_results):
            fusion_scores[idx] = fusion_scores.get(idx, 0) + 0.3 * (1 - i/top_k)
        for i, doc_id in enumerate(vector_results['ids'][0]):
            idx = int(doc_id)
            fusion_scores[idx] = fusion_scores.get(idx, 0) + 0.7 * (1 - i/top_k)
        
        sorted_results = sorted(fusion_scores.items(), key=lambda x: x[1], reverse=True)
        return [{"text": self.corpus[idx], "score": score} for idx, score in sorted_results[:top_k]]
```
