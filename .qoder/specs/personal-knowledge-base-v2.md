# 第一部分：项目概述

## 1. 项目背景与目标

### 1.1 项目背景

在信息爆炸的时代，个人每天产生大量的思考、想法、学习成果和决策记录。传统工具（笔记软件、文档系统）存在以下核心问题：

| 问题 | 描述 | 影响 |
|------|------|------|
| 知识孤岛 | 信息分散在多个平台，无法关联 | 知识利用率低 |
| 一次性答案 | 每次提问都需重新检索、分析 | 重复劳动 |
| 维护成本高 | 需要人工整理、归类、更新 | 难以坚持 |
| 技术依赖 | 商业服务可能停止，数据可能被锁定 | 长期风险 |

### 1.2 项目目标

本项目旨在构建一套**个人知识库系统 (Personal Knowledge Base, PKB)**，实现以下核心目标：

1. **知识持续沉淀**：不是一次性问答，而是让知识随着时间不断积累、结构化
2. **多端便捷输入**：通过飞书、CLI等工具随时随地记录
3. **长期稳定存储**：不受技术迭代影响，10年以上可用
4. **智能分析能力**：AI辅助理解和关联，发现隐藏的知识连接

## 2. 设计理念：LLM Wiki

本系统借鉴 Andrej Karpathy 提出的 **LLM Wiki** 理念，重新定义 AI 在知识管理中的角色：

> 传统模式：LLM 作为"问答器" → 提问时临时检索 → 一次性答案
>
> LLM Wiki 模式：LLM 作为"Wiki维护者" → 平时持续维护 → 可积累的结构化资产

**核心转变**：让 LLM 承担低创造性、高维护成本的工作（整理、归档、链接、更新），人类专注于判断什么重要。

## 3. 术语表

|------|------|
| PKB | Personal Knowledge Base，个人知识库系统 |

### 4.2 用户故事

#### US-001: 快速记录笔记 [P0]

**作为** 记录者，**我想要** 通过飞书私聊发送一条消息并添加 `#note` 指令，**以便** 快速将想法存入知识库。

- **验收标准**：
  - 消息在 3 秒内被系统接收并存储到 `raw/inbox/`
  - LLM 在后台异步分析内容并更新相关 Wiki 页面
  - 用户收到确认回复，包含已处理的摘要
- **边界情况**：
  - 消息超过 4000 字符时自动分段处理
  - 网络中断时消息缓存到本地，恢复后自动同步
  - 重复内容（哈希相同）不重复存储

#### US-002: 语义搜索知识 [P0]

**作为** 询问者，**我想要** 使用自然语言搜索知识库，**以便** 找到相关的历史信息和认知。

- **验收标准**：
  - 搜索结果在 2 秒内返回（本地模型）或 5 秒内（云端模型）
  - 结果包含 Wiki 页面引用和来源链接
  - 支持 `#search` 指令在飞书和 CLI 中使用
- **边界情况**：
  - 搜索词为空时返回最近更新的 10 条内容
  - 无匹配结果时提供近似建议和搜索提示
  - 特殊字符和 Emoji 正常处理

#### US-003: 命令行管理知识 [P0]

**作为** 记录者，**我想要** 通过 CLI 工具添加、搜索、列表和管理笔记，**以便** 在终端环境中高效操作。

- **验收标准**：
  - `pkb add` 命令在 2 秒内完成存储
  - `pkb search` 返回格式化的搜索结果
  - `pkb list --tag <tag>` 按标签过滤
  - 支持 `--file` 和 `--url` 参数添加外部内容
- **边界情况**：
  - API 服务未启动时提示并尝试自动启动
  - 文件超过 10MB 时提示使用文件上传接口
  - 无效 URL 时返回明确的错误信息

#### US-004: 上传文档并自动处理 [P1]

**作为** 记录者，**我想要** 上传 PDF/DOCX/MD 文件，**以便** 系统自动解析内容并整合到知识库。

- **验收标准**：
  - 支持 PDF、DOCX、Markdown、TXT 格式
  - 文件解析成功后生成摘要和关键信息提取
  - 原始文件保存到 `raw/documents/`，解析结果更新 Wiki
- **边界情况**：
  - 加密 PDF 提示用户输入密码或跳过
  - 扫描版 PDF 提示需要 OCR 处理（标记为待处理）
  - 超大文件（>50MB）异步处理并通知完成状态

#### US-005: 给内容添加标签 [P1]

**作为** 记录者，**我想要** 给笔记和文档添加标签，**以便** 后续按类别浏览和检索。

- **验收标准**：
  - `#tag` 指令在飞书中添加标签到最近内容
  - `pkb tag <id> <tags>` 在 CLI 中添加标签
  - AI 自动推荐合适的标签
- **边界情况**：
  - 标签不存在时自动创建
  - 标签名称包含特殊字符时自动清理
  - 批量添加标签时支持逗号分隔

#### US-006: 查看系统状态 [P1]

**作为** 审计者，**我想要** 查看系统运行状态、存储使用和最近操作，**以便** 确保系统健康运行。

- **验收标准**：
  - `#status` 指令返回存储统计、服务状态、最近操作
  - `pkb status` 在 CLI 中显示格式化状态信息
  - 包含磁盘使用率、文档数量、Wiki 页面数量
- **边界情况**：
  - 服务部分异常时显示降级状态
  - 磁盘使用率超过 85% 时发出警告

#### US-007: 自动巡检维护 Wiki 质量 [P1]

**作为** 维护者，**我想要** 系统每周自动执行巡检，**以便** 发现孤儿页、重复内容和过期信息。

- **验收标准**：
  - 每周自动执行 Lint 巡检
  - 检测孤儿页、重复页、过期标记、死链
  - 生成巡检报告并记录到 `wiki/log.md`
  - 支持手动触发 `pkb lint`
- **边界情况**：
  - 巡检过程中发现严重问题时暂停并通知用户
  - 巡检耗时超过 30 分钟时自动暂停，下次继续
  - 用户可在巡检期间手动修改页面（乐观锁处理）

#### US-008: 知识回写到 Wiki [P1]

**作为** 询问者，**我想要** 将有价值的问答结果自动沉淀为新的 Wiki 页面，**以便** 知识持续积累。

- **验收标准**：
  - 回答包含新认知时，系统建议创建新 Wiki 页面
  - 用户确认后自动创建页面并补全交叉链接
  - 新页面遵循标准格式（frontmatter + 内容结构）
- **边界情况**：
  - 新页面与现有页面高度相似时提示合并
  - 用户拒绝回写时记录反馈用于优化判断逻辑

#### US-009: 管理 Wiki 页面 [P2]

**作为** 维护者，**我想要** 手动创建、编辑和删除 Wiki 页面，**以便** 精确控制知识结构。

- **验收标准**：
  - API 支持 CRUD 操作于 Wiki 页面
  - 编辑时保留版本历史
  - 删除时检查是否有其他页面引用
- **边界情况**：
  - 删除被引用页面时提示确认或自动更新引用
  - 编辑冲突时提供合并选项

#### US-010: 备份和恢复数据 [P1]

**作为** 审计者，**我想要** 定期备份数据并在需要时恢复，**以便** 防止数据丢失。

- **验收标准**：
  - 支持每日增量备份和每周完整备份
  - 关键数据实时同步到云端（Cloudflare R2）
  - 恢复时间 < 30 分钟
- **边界情况**：
  - 备份过程中磁盘空间不足时暂停并通知
  - 恢复时校验数据完整性（checksum 验证）

#### 5.1.1 飞书 Webhook 响应机制

飞书事件回调要求在 **3 秒内返回 HTTP 200**，否则会重试（最多 3 次）。因此所有耗时操作必须异步处理：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   飞书 Webhook 响应时序                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   飞书服务器                    PKB 服务                                  │
│       │                          │                                       │
│       │  1. POST /webhook/event  │                                       │
│       │  (im.message.receive_v1) │                                       │
│       │ ───────────────────────> │                                       │
│       │                          │  2. 验证签名 + 去重                    │
│       │                          │  3. 提交任务到 task_queue              │
│       │                          │                                       │
│       │  4. HTTP 200 (立即返回)   │                                       │
│       │ <─────────────────────── │  ← 整个处理 < 500ms                   │
│       │                          │                                       │
│       │                          │  5. 后台 Worker 异步执行:              │
│       │                          │     ├── Ingest Pipeline               │
│       │                          │     └── Query Pipeline                │
│       │                          │                                       │
│       │                          │  6. 完成后主动发送飞书消息              │
│       │  7. 回复消息给用户        │     (POST /im/v1/messages)            │
│       │ <─────────────────────── │                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**关键设计**：

| 环节 | 要求 | 实现方式 |
|------|------|----------|
| 签名验证 | < 50ms | 缓存 Verification Token，HMAC-SHA256 |
| 事件去重 | < 100ms | 内存 LRU 缓存（最近 1000 个 event_id，5 分钟 TTL） |
| 任务提交 | < 200ms | 写入 SQLite task_queue 表 |
| **总响应时间** | **< 500ms** | 确保远在 3 秒超时内 |

**事件去重实现**：

```python
# backend/src/feishu/dedup.py
from collections import OrderedDict

class EventDeduplicator:
    """飞书事件去重，防止重试导致重复处理"""

    def __init__(self, max_size: int = 1000, ttl_seconds: int = 300):
        self._cache: OrderedDict[str, float] = OrderedDict()
        self._max_size = max_size
        self._ttl = ttl_seconds

    def is_duplicate(self, event_id: str) -> bool:
        """检查事件是否已处理"""
        now = time.time()

        # 清理过期条目
        expired = [k for k, v in self._cache.items() if now - v > self._ttl]
        for k in expired:
            del self._cache[k]

        if event_id in self._cache:
            return True

        # 记录新事件
        self._cache[event_id] = now
        if len(self._cache) > self._max_size:
            self._cache.popitem(last=False)  # 移除最旧的

        return False
```

**飞书事件处理流程**：

```python
# backend/src/feishu/handlers.py
from fastapi import APIRouter, Request, Response

router = APIRouter()
dedup = EventDeduplicator()

@router.post("/webhook/event")
async def handle_feishu_event(request: Request):
    """飞书事件回调处理"""

    # 1. 读取请求体
    body = await request.body()
    headers = request.headers

    # 2. 验证签名（确保请求来自飞书）
    if not verify_signature(body, headers):
        return Response(status_code=401)

    # 3. 解析事件
    event_data = json.loads(body)

    # 3.1 处理 URL 验证（首次配置时）
    if event_data.get("type") == "url_verification":
        return {"challenge": event_data["challenge"]}

    # 4. 事件去重
    event_id = event_data.get("header", {}).get("event_id", "")
    if dedup.is_duplicate(event_id):
        return Response(status_code=200)  # 静默忽略重复事件

    # 5. 提交到任务队列（异步处理）
    await submit_task(
        task_type="ingest",
        payload={
            "source": "feishu",
            "event_id": event_id,
            "event_data": event_data,
        },
        priority=7,  # 飞书消息优先级较高
    )

    # 6. 立即返回 200（< 500ms）
    return Response(status_code=200)
```

async def send_reply(chat_id: str, message_id: str, text: str):

### 9.1 性能

> **设计原则**：个人知识库以**质量优先**，非高并发场景，性能指标应贴合实际使用模式。

| 指标 | 要求 | 场景分类 | 说明 |
|------|------|----------|------|
| **搜索响应时间** | | | |
| ├ 关键词搜索 | < 100ms | SQLite FTS | 纯文本匹配 |
| ├ 向量检索 | < 500ms | ChromaDB top-5 | 语义搜索 |
| └ LLM 综合回答 | 1-5s | 含模型推理 | 本地 Ollama 500ms-2s，云端 1-3s |
| **并发能力** | | | |
| ├ 日常使用 | 1-2 QPS | 单人交互 | 个人场景足够 |
| ├ 后台处理 | 串行执行 | Ingest/Lint | 避免资源竞争 |
| └ 峰值容忍 | 5 QPS / 30s | 批量导入 | 短时间突发 |
| **启动时间** | | | |
| ├ 服务冷启动 | < 10s | FastAPI + DB 初始化 | 首次加载模型除外 |
| └ Ollama 模型加载 | 5-15s | 首次推理请求 | 模型驻留后无延迟 |
| **索引速度** | | | |
| ├ 单条 Ingest | 1-3s | 含 embedding + LLM 分析 | 异步处理 |
| ├ 批量导入 | 50-100条/分钟 | 后台队列 | 取决于 embedding 模型 |
| └ Lint 巡检 | 10页/分钟 | 质量检查 | 本地模型 |

**性能优化策略**：
1. **异步处理**：Ingest Pipeline 使用后台队列，不阻塞用户输入
2. **缓存机制**：热门搜索结果缓存 5 分钟，避免重复 embedding 计算
3. **模型驻留**：Ollama 模型加载后保持内存，避免重复加载延迟
4. **并发限制**：使用 Semaphore 限制同时 LLM 请求数（建议 2），防止 OOM

### 12.3 任务队列

PKB 使用**内置 asyncio 队列**作为任务调度机制，不引入外部消息队列（个人场景无需 Celery/Redis），通过 SQLite 持久化保证任务不丢失。

#### 12.3.1 架构设计

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       任务队列架构                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌────────────────────────────────────────────────────────────────┐   │
│   │                    FastAPI 请求线程                              │   │
│   │   POST /messages  →  写入 task_queue 表  →  返回 task_id       │   │
│   │   POST /lint      →  写入 task_queue 表  →  返回 task_id       │   │
│   └───────────────────────────┬────────────────────────────────────┘   │
│                               │                                         │
│                               ▼                                         │
│   ┌────────────────────────────────────────────────────────────────┐   │
│   │                 asyncio 后台 Worker                              │   │
│   │                                                                 │   │
│   │   TaskWorker (单进程，串行执行)                                  │   │
│   │   ├── 轮询 task_queue 表 (pending 状态)                          │   │
│   │   ├── 取出任务 → 标记 processing                                 │   │
│   │   ├── 执行对应 Pipeline                                         │   │
│   │   ├── 成功 → 标记 completed + 记录结果                           │   │
│   │   └── 失败 → 标记 failed + 记录错误 + 重试计数                   │   │
│   │                                                                 │   │
│   │   并发控制:                                                     │   │
│   │   ├── Ingest: Semaphore(2) — 最多 2 个并行                      │   │
│   │   ├── Lint:   Semaphore(1) — 串行，避免资源竞争                 │   │
│   │   └── Query:  Semaphore(3) — 最多 3 个并行                      │   │
│   └────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   ┌────────────────────────────────────────────────────────────────┐   │
│   │                 task_queue 表 (SQLite 持久化)                    │   │
│   │                                                                 │   │
│   │   id | task_type | status | payload | result | retries | ...    │   │
│   │                                                                 │   │
│   │   状态: pending → processing → completed/failed                 │   │
│   │   重试: 最多 3 次，指数退避 (1s → 2s → 4s)                     │   │
│   └────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

#### 12.3.3 任务 Worker 实现

```python
# backend/src/core/task_worker.py
import asyncio
import json
from datetime import datetime, timedelta, timezone

class TaskWorker:
    """后台任务 Worker，串行执行 Ingest/Lint，并行执行 Query"""

    def __init__(self, db_session_factory, poll_interval: float = 1.0):
        self.db_session_factory = db_session_factory
        self.poll_interval = poll_interval
        self._running = False
        self._semaphores = {
            "ingest": asyncio.Semaphore(2),
            "query": asyncio.Semaphore(3),
            "lint": asyncio.Semaphore(1),
            "backup": asyncio.Semaphore(1),
            "reindex": asyncio.Semaphore(1),
        }
        self._handlers = {
            "ingest": self._handle_ingest,
            "query": self._handle_query,
            "lint": self._handle_lint,
            "backup": self._handle_backup,
            "reindex": self._handle_reindex,
        }

    async def start(self):
        """启动 Worker 主循环"""
        self._running = True
        while self._running:
            await self._poll_and_execute()
            await asyncio.sleep(self.poll_interval)

    async def stop(self):
        """优雅停止"""
        self._running = False

    async def _poll_and_execute(self):
        """轮询数据库获取待执行任务"""
        async with self.db_session_factory() as session:
            task = await self._fetch_pending_task(session)
            if not task:
                return

            sem = self._semaphores.get(task["task_type"])
            if sem:
                async with sem:
                    await self._execute_task(session, task)

    async def _execute_task(self, session, task: dict):
        """执行单个任务"""
        # 标记为 processing
        await self._update_status(session, task["id"], "processing")

        try:
            handler = self._handlers.get(task["task_type"])
            if not handler:
                raise ValueError(f"未知任务类型: {task['task_type']}")

            result = await handler(json.loads(task["payload"]))

            # 成功：记录结果
            await self._update_status(
                session, task["id"], "completed",
                result=json.dumps(result, ensure_ascii=False)
            )
        except Exception as e:
            # 失败：判断是否重试
            retries = task["retries"] + 1
            if retries < task["max_retries"]:
                next_retry = datetime.now(timezone.utc) + timedelta(
                    seconds=2 ** retries  # 指数退避: 2s → 4s → 8s
                )
                await self._update_status(
                    session, task["id"], "pending",
                    retries=retries,
                    next_retry_at=next_retry.isoformat(),
                    error_message=str(e),
                )
            else:
                await self._update_status(
                    session, task["id"], "failed",
                    error_message=str(e),
                )

    async def _handle_ingest(self, payload: dict) -> dict:
        """Ingest 任务处理"""
        from backend.src.services.ingest import ingest_document
        return await ingest_document(**payload)

    async def _handle_query(self, payload: dict) -> dict:
        """Query 任务处理"""
        from backend.src.services.query import query_knowledge
        return await query_knowledge(**payload)

    async def _handle_lint(self, payload: dict) -> dict:
        """Lint 任务处理"""
        from backend.src.services.lint import run_lint
        return await run_lint(**payload)

    async def _handle_backup(self, payload: dict) -> dict:
        """Backup 任务处理"""
        from backend.src.core.file_storage import run_backup
        return await run_backup(**payload)

    async def _handle_reindex(self, payload: dict) -> dict:
        """Reindex 任务处理"""
        from backend.src.core.vector_store import rebuild_index
        return await rebuild_index(**payload)
```

```python
# backend/src/core/task_queue.py
async def submit_task(
    task_type: str,
    payload: dict,
    priority: int = 5,
) -> str:
    """提交任务到队列，返回 task_id"""
    task_id = generate_uuid()
    async with get_db_session() as session:
        session.add(TaskQueue(
            id=task_id,
            task_type=task_type,
            status="pending",
            priority=priority,
            payload=json.dumps(payload, ensure_ascii=False),
        ))
        await session.commit()
    return task_id


async def get_task_status(task_id: str) -> dict | None:
    """查询任务状态"""
    async with get_db_session() as session:
        task = await session.get(TaskQueue, task_id)
        if not task:
            return None
        return {
            "task_id": task.id,
            "task_type": task.task_type,
            "status": task.status,
            "result": json.loads(task.result) if task.result else None,
            "error_message": task.error_message,
            "retries": task.retries,
            "created_at": task.created_at.isoformat(),
            "completed_at": task.completed_at.isoformat() if task.completed_at else None,
        }
```

| 任务类型 | 默认优先级 | 并发度 | 说明 |
|----------|-----------|--------|------|

| `query` | 7 | 3 | 用户搜索，优先级最高 |
| `lint` | 3 | 1 | 后台维护，低优先级 |

服务重启时，Worker 自动恢复未完成的任务：

async def recover_stale_tasks():
    """恢复重启前未完成的任务"""
    async with get_db_session() as session:
        # 将 processing 状态的任务重置为 pending（服务重启意味着执行中断）
        await session.execute(
            update(TaskQueue)
            .where(TaskQueue.status == "processing")
            .values(status="pending", retries=0)

        await session.commit()

const apiClient = axios.create({
    baseURL: '/api/v1',

});

apiClient.interceptors.request.use((config) => {
    const token = localStorage.getItem('pkb_token');
    if (token) {
        config.headers.Authorization = `Bearer ${token}`;
    }
    return config;

apiClient.interceptors.response.use(
    (response) => response,
    (error) => {
        if (error.response?.status === 401) {
            // Token 无效，跳转登录
            localStorage.removeItem('pkb_token');
            window.location.href = '/login';
        }
        return Promise.reject(error);

## 14. API 设计原则

### 14.1 设计哲学

PKB API 遵循以下设计原则：

1. **RESTful 风格**：资源导向的 URL 设计，使用标准 HTTP 方法
2. **版本控制**：所有接口以 `/api/v1/` 为前缀，便于未来版本迭代
3. **JSON 优先**：请求和响应统一使用 JSON 格式
4. **异步处理**：耗时操作（Ingest、Lint）返回任务 ID，支持状态轮询
5. **错误一致**：所有错误响应遵循统一格式
6. **认证简化**：个人项目使用 API Token 认证，无需 OAuth

| 422 | Unprocessable Entity | 请求格式正确但语义错误 |
| 429 | Too Many Requests | 超过速率限制 |
| 500 | Internal Server Error | 服务器内部错误 |

#### 14.2.6 速率限制

| 接口类别 | 限制 | 时间窗口 |
|----------|------|----------|
| 消息接口 | 30 次 | 1 分钟 |
| 搜索接口 | 20 次 | 1 分钟 |
| 文档接口 | 10 次 | 1 分钟 |
| Wiki 接口 | 20 次 | 1 分钟 |
| 系统接口 | 5 次 | 1 分钟 |

超过限制时返回 `429 Too Many Requests`：

```json
{
    "success": false,
    "error": {
        "code": "RATE_LIMITED",
        "message": "请求过于频繁，请稍后重试",
        "details": {
            "retry_after": 30
        }
    }
}
```

1. 中文标题：使用 LLM 生成简洁英文 slug（如 "AI与机器学习" → "ai-and-ml"）

4. Sources 页面：使用日期 + 序号（如 "2026-04-18-feishu-001"）
5. 唯一性保证：slug 冲突时追加数字后缀（如 "ai-and-ml-2"）

#### 20.1.2 表结构定义

```sql
-- 文档表
CREATE TABLE documents (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    content_hash TEXT UNIQUE,
    source TEXT NOT NULL CHECK(source IN ('feishu', 'cli', 'web', 'file')),
    source_id TEXT,
    doc_type TEXT CHECK(doc_type IN ('note', 'doc', 'web', 'chat')),
    file_path TEXT,
    file_size INTEGER,
    language TEXT DEFAULT 'zh-CN',
    word_count INTEGER DEFAULT 0,
    summary TEXT,
    vector_id TEXT,
    status TEXT DEFAULT 'active' CHECK(status IN ('active', 'archived', 'deleted')),
    is_favorite INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME,
    deleted_at DATETIME,

    INDEX idx_source (source),
    INDEX idx_created (created_at),
    INDEX idx_status (status)
);

-- 记忆表
CREATE TABLE memories (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL CHECK(type IN ('thought', 'idea', 'learn', 'todo', 'note', 'question')),
    content TEXT NOT NULL,
    embedding_id TEXT,
    parent_id TEXT REFERENCES memories(id),
    doc_id TEXT REFERENCES documents(id),
    importance INTEGER DEFAULT 3 CHECK(importance BETWEEN 1 AND 5),
    mood TEXT,
    language TEXT DEFAULT 'zh-CN',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME,

    INDEX idx_type (type),
    INDEX idx_created (created_at),
    INDEX idx_parent (parent_id)
);

-- 实体表 (知识图谱)
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    type TEXT NOT NULL CHECK(type IN ('person', 'project', 'concept', 'tool', 'location', 'event', 'organization')),
    description TEXT,
    aliases TEXT,  -- JSON array
    properties TEXT,  -- JSON object
    first_seen DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_seen DATETIME,
    mention_count INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(name, type),
    INDEX idx_type (type),
    INDEX idx_name (name)
);

-- 实体关系表
CREATE TABLE entity_relations (
    id TEXT PRIMARY KEY,
    source_id TEXT NOT NULL REFERENCES entities(id),
    target_id TEXT NOT NULL REFERENCES entities(id),
    relation_type TEXT NOT NULL CHECK(relation_type IN ('works_on', 'uses', 'similar_to', 'part_of', 'depends_on', 'related_to')),
    properties TEXT,  -- JSON object
    confidence REAL DEFAULT 1.0 CHECK(confidence BETWEEN 0 AND 1),
    source_text TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(source_id, target_id, relation_type),
    INDEX idx_source (source_id),
    INDEX idx_target (target_id)
);

-- Wiki页面表
CREATE TABLE wiki_pages (
    id TEXT PRIMARY KEY,
    path TEXT UNIQUE NOT NULL,
    slug TEXT UNIQUE NOT NULL,
    wiki_type TEXT NOT NULL CHECK(wiki_type IN ('sources', 'topics', 'entities', 'projects', 'decisions')),
    title TEXT NOT NULL,
    content TEXT,
    doc_id TEXT REFERENCES documents(id),
    version INTEGER NOT NULL DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME,

    INDEX idx_type (wiki_type),
    INDEX idx_path (path),
    INDEX idx_slug (slug)
);

-- Wiki页面版本历史表
CREATE TABLE wiki_page_versions (
    id TEXT PRIMARY KEY,
    page_id TEXT NOT NULL REFERENCES wiki_pages(id),
    version INTEGER NOT NULL,
    title TEXT NOT NULL,
    content TEXT NOT NULL,

### 20.2 Wiki 页面版本控制

#### 20.2.1 版本控制策略

Wiki 页面采用**乐观锁 + 快照版本**机制，每次更新自动创建历史快照：

| 机制 | 说明 |
|------|------|
| 乐观锁 | 更新时必须提供当前 `version`，不匹配则返回 409 Conflict |
| 快照存储 | 每次更新前将当前内容完整写入 `wiki_page_versions` |
| 变更溯源 | `changed_by` 字段区分变更来源（LLM Ingest / LLM Lint / 用户手动） |
| 版本上限 | 每页保留最近 50 个版本，超出后自动清理最旧版本 |
| 自动摘要 | LLM 更新时自动生成 `change_summary`，用户更新时可选填写 |

#### 20.2.2 版本控制流程

```

#### 20.2.3 版本控制代码模式

```python
# 模式：乐观锁更新 + 版本快照
async def update_wiki_page(
    page_id: str,
    new_content: str,
    expected_version: int,
    changed_by: str,
    change_summary: str | None = None,
) -> WikiPage:
    """带版本控制的 Wiki 页面更新"""

    async with get_db_session() as session:
        # 1. 读取当前页面
        page = await session.get(WikiPage, page_id)
        if not page:
            raise NotFoundError(f"页面不存在: {page_id}")

        # 2. 乐观锁检查
        if page.version != expected_version:
            raise VersionConflictError(
                current=page.version,
                provided=expected_version
            )

        # 3. 快照当前版本到历史表
        version_snapshot = WikiPageVersion(
            id=generate_uuid(),
            page_id=page.id,
            version=page.version,
            title=page.title,
            content=page.content,
            frontmatter=extract_frontmatter(page.content),
            change_summary=change_summary or f"更新前快照 v{page.version}",
            changed_by=changed_by,
        )
        session.add(version_snapshot)

        # 4. 写入新内容
        page.content = new_content
        page.version += 1
        page.updated_at = utcnow()

        # 5. 清理超限的旧版本（保留最近 50 个）
        old_versions = await session.execute(
            select(WikiPageVersion)
            .where(WikiPageVersion.page_id == page_id)
            .order_by(WikiPageVersion.version.desc())
            .offset(50)
        )
        for old_ver in old_versions.scalars():
            await session.delete(old_ver)

        await session.commit()
        return page


async def revert_wiki_page(
    page_id: str,
    target_version: int,
    changed_by: str,
) -> WikiPage:
    """回滚 Wiki 页面到指定版本"""

    async with get_db_session() as session:
        page = await session.get(WikiPage, page_id)
        if not page:
            raise NotFoundError(f"页面不存在: {page_id}")

        # 读取目标版本内容
        target = await session.execute(
            select(WikiPageVersion)
            .where(
                WikiPageVersion.page_id == page_id,
                WikiPageVersion.version == target_version
            )
        )
        target_ver = target.scalar_one_or_none()
        if not target_ver:
            raise NotFoundError(f"版本不存在: v{target_version}")

        # 复用更新流程（快照当前 + 写入目标内容）
        return await update_wiki_page(
            page_id=page_id,
            new_content=target_ver.content,
            expected_version=page.version,
            changed_by=changed_by,
            change_summary=f"回滚到版本 v{target_version}",
        )
```

#### 23.2.3 Embedding 降级策略

```

  新内容 → 检查 OpenAI 可用 → 可用则用 openai → 不可用则用 local

降级模式1 (网络中断):
  新内容 → 直接使用 local embedding → 记录降级日志

降级模式2 (预算超限):
  新内容 → 自动切换到 local → 通知用户

# 模式：LLM 输出验证
def validate_llm_wiki_update(llm_output: dict) -> ValidationResult:
    """验证 LLM 生成的 Wiki 更新是否合理"""
    issues = []

    # 检查1：引用来源是否存在
    for ref in llm_output.get("source_refs", []):
        if not wiki_page_exists(ref):
            issues.append(f"引用页面不存在: {ref}")

    # 检查2：内容长度是否合理
    if len(llm_output.get("content", "")) > 10000:
        issues.append("内容过长，可能包含幻觉")

    # 检查3：置信度阈值
    if llm_output.get("confidence", 0) < 0.7:
        issues.append("置信度过低，需人工审核")

    return ValidationResult(
        passed=len(issues) == 0,
        issues=issues
    )

## 27. 错误处理

### 27.1 错误分类与处理

> **设计原则**：个人知识库以**数据完整性优先**，任何错误都不应导致数据丢失。

| 错误类型 | 严重程度 | 示例场景 | 处理策略 | 用户通知 |
|----------|----------|----------|----------|----------|
| **数据层错误** | | | | |
| SQLite 损坏 | P0 | 数据库文件损坏/锁定 | 1. 立即停止写入<br>2. 从最近备份恢复<br>3.  WAL 回滚 | 告警 + 停止服务 |
| ChromaDB 异常 | P1 | 向量索引损坏 | 1. 重建索引（从 raw 重新 embedding）<br>2. 标记问题文档 | 告警 + 降级服务 |
| 文件系统错误 | P1 | 磁盘满/权限问题 | 1. 检查磁盘空间<br>2. 清理临时文件<br>3. 修复权限 | 告警 |
| **LLM 服务错误** | | | | |
| Ollama 超时 | P2 | 模型推理 > 30s | 1. 重试 2 次<br>2. 降级为关键词搜索<br>3. 记录失败日志 | 静默降级 |
| 云端 API 失败 | P2 | OpenAI 网络错误/配额用完 | 1. 重试 3 次（指数退避）<br>2. 切换到本地模型<br>3. 提示用户充值 | 提示切换 |
| LLM 幻觉 | P1 | 生成不存在的内容 | 1. 引用来源验证<br>2. 置信度 < 0.7 时标记<br>3. 用户反馈机制 | 标记不确定 |
| **Ingest 管线错误** | | | | |
| 文档解析失败 | P2 | PDF 格式不支持 | 1. 保存原始文件到 raw/inbox<br>2. 标记解析失败<br>3. 手动处理队列 | 提示手动处理 |
| Embedding 失败 | P2 | 文本过长/编码问题 | 1. 截断或分块重试<br>2. 跳过问题段落<br>3. 记录失败统计 | 静默处理 |
| Wiki 更新冲突 | P1 | 并发修改同一页面 | 1. 乐观锁重试<br>2. 合并冲突解决<br>3. 保留两个版本 | 提示冲突 |
| **外部服务错误** | | | | |
| 飞书 API 异常 | P2 | 消息发送失败/webhook 超时 | 1. 重试 3 次<br>2. 本地消息队列缓存<br>3. 恢复后补发 | 提示延迟 |
| 网络中断 | P1 | 完全断网 | 1. 本地缓存所有输入<br>2. 断网模式可用 CLI<br>3. 恢复后同步 | 提示离线 |

### 27.2 错误处理代码模式

```python
# 模式1：关键操作的事务保护
async def ingest_document(content: str):
    try:
        # 1. 保存原始数据（事务）
        raw_path = await save_to_raw(content)

        # 2. 向量化（可重试）
        embedding = await retry_with_backoff(
            create_embedding,
            max_retries=3,
            content=content
        )

        # 3. LLM 分析（可降级）
        try:
            analysis = await llm_analyze(content, timeout=30)
        except LLMTimeoutError:
            analysis = await fallback_to_local_llm(content)

        # 4. 更新 Wiki（乐观锁）
        await update_wiki_with_retry(analysis, max_retries=2)

    except CriticalError as e:
        # 回滚：删除已创建的临时文件
        await rollback_ingest(raw_path)
        raise
    finally:
        # 无论成功失败，记录审计日志
        await audit_log("ingest", status=success_or_fail)
```

### 27.3 降级策略

降级模式2（向量库损坏）:

#### 27.5.1 回滚场景与策略

#### 27.5.2 常用恢复命令

**场景1：恢复误删的 Wiki 页面**

```bash

$ cp backups/weekly/2026-W16/data/wiki/topics/ai-and-ml.md data/wiki/topics/

```

**场景2：SQLite 数据库损坏恢复**

```bash
# 步骤1：检查 WAL 文件（自动回滚）
$ sqlite3 data/db/pkb.db "PRAGMA integrity_check;"

# 如果损坏，尝试 WAL 回滚
$ sqlite3 data/db/pkb.db "PRAGMA wal_checkpoint(TRUNCATE);"

# 步骤2：如果 WAL 回滚失败，从备份恢复
$ mv data/db/pkb.db data/db/pkb.db.corrupted  # 备份损坏文件
$ cp backups/weekly/2026-W16/data/db/pkb.db data/db/

# 步骤3：验证恢复
$ sqlite3 data/db/pkb.db "PRAGMA integrity_check;"

# 步骤4：重建向量索引（如果需要）
$ pkb reindex --all
```

#### 28.1.5 成本优化建议

1. **优先本地模型**：90% 场景使用 Ollama，仅复杂推理用云端

## 附录B: AGENTS.md 模板

```markdown
# PKB AGENTS.md

## 系统角色

你是一个个人知识库维护助手 (Wiki Maintainer)。
你的职责是持续维护一个结构化的个人Wiki，而非仅仅回答问题。

## 目录规则

```
root/
├── raw/           # 原始资料层，source of truth，LLM不可修改
│   ├── inbox/     # 新进入的资料
│   ├── notes/    # 日常随手记
│   ├── documents/# 上传的文档
│   └── feishu/   # 飞书归档
│
├── wiki/          # Wiki层，LLM维护
│   ├── index.md  # 知识库总览
│   ├── log.md    # 操作日志
│   ├── sources/  # 资料索引
│   ├── topics/   # 主题
│   ├── entities/ # 实体
│   ├── projects/ # 项目
│   └── decisions/ # 决策记录
│
└── CONFIG/        # 配置文件
    └── templates/ # 页面模板
```

## Ingest 规则

当新资料进入时，执行以下步骤：

1. **读取原始资料**
   - 完整理解内容
   - 提取关键信息

2. **创建/更新 Sources 页**
   - 在 `wiki/sources/` 创建对应页面
   - 包含：来源、日期、关键结论、相关页面

3. **分析关联**
   - 找出相关的 topics/entities/projects 页面
   - 如无对应页面，建议创建

4. **更新 Wiki 页面**
   - 在相关页面添加信息
   - 补全交叉链接

5. **更新索引**
   - 更新 `wiki/index.md` 的最近更新列表
   - 如有必要，更新标签

6. **记录日志**
   - 追加到 `wiki/log.md`

## Query 规则

当用户提问时，执行以下步骤：

1. **理解问题**
   - 明确查询意图
   - 确定相关主题

2. **定位 Wiki 页面**
   - 先读 `wiki/index.md`
   - 定位相关页面

3. **综合回答**
   - 基于 Wiki 页面回答
   - 引用来源链接
   - 不确定的内容明确说明

4. **评估回写价值**
   - 如果回答包含新认知
   - 建议创建/更新 Wiki 页面

## Lint 规则

每周巡检时检查：

1. **孤儿页检测**
   - 找出未被引用的页面
   - 决定：保留、合并还是归档

2. **重复检测**
   - 找出内容相似的页面
   - 合并重复内容

3. **过期标记**
   - 检测过时结论
   - 标记或更新

4. **链接检查**
   - 修复死链
   - 补全缺失链接

5. **概念提升**
   - 找出高频提及但无页面的概念
   - 建议创建新页面

## 页面规范

### 必须字段

每个 Wiki 页面必须包含：

```yaml
---
id: <uuid>
type: <topics|entities|projects|decisions>
slug: <ascii-slug>
title: <页面标题>
created_at: <ISO时间>
updated_at: <ISO时间>
tags: [<标签列表>]
related_pages:
  - <相关页面路径>
source_ref:
  - <原始资料路径>
---
```

### 页面类型特定要求

**topics/**:
- 定义
- 关键概念
- 相关项目
- 最近认知

**entities/**:
- 简介
- 类型
- 使用场景
- 相关决策

**projects/**:
- 概述
- 技术栈
- 关键决策
- 进展状态

**decisions/**:
- 背景
- 选项
- 最终决定
- 理由

## 日志规范

每次 Ingest/Lint 后追加到 `wiki/log.md`:

```markdown
## 2026-04-18

### Ingest
- 10:30 - 处理飞书消息 → 创建 [[wiki/sources/2026-04-18-xxx]]
- 10:32 - 更新 [[wiki/topics/ai-and-ml]]

### Lint
- 14:00 - 巡检完成
  - 发现1个孤儿页：xxx
  - 合并重复页面：xxx

### Query
- 15:00 - 回答"如何配置Ollama"
  - 来源：[[wiki/topics/ollama-config]]
  - 建议创建：[[wiki/decisions/2026-04-ollama-selection]]
```

## 约束

1. **不撒谎**: 不确定的内容明确说"我不知道"
2. **引用来源**: 所有观点注明来源
3. **保持简洁**: 避免冗余内容
4. **维护一致性**: 定期检查页面间的一致性
```
